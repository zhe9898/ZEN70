# 0007: M8 家族信标采用纯 SSE 与原生 SQLAlchemy (Native Family Board via SSE & RLS)

- **状态**: 接受
- **日期**: 2026-03-17

## 1. 背景上下文
为了填补 ZEN70 核心体系中缺乏的多端信息协同与“留言板”特性，我们引入了 Feature M8 家族信标。开发过程中，需要评估技术选型以最低的代码代价和最高的运行效能支持瀑布流留言及实时双向交互推送。

## 2. 决策选项
1. **方案 A**: 部署由 Next.js / Node 编写的外部第三方独立协同套件。
2. **方案 B**: 在 FastAPI 内自建 WebSockets 持久保持连接，进行全双工收发。
3. **方案 C**: 借力原生系统已存在的 Redis Pub/Sub 与 FastAPI 底层生成器，通过 `Server-Sent Events (SSE)` 的单向广播技术加上纯净的单向 REST POST 实现通信。

## 3. 评估对比
### 方案 A (第三方隔离应用)
- **优势**: 有现成的套件可直接挂载。
- **劣势**: 造成运维极大割裂（不互通的 Auth），并且违规《法典 1.1》，会极大增加整体应用包袱。

### 方案 B (WebSockets)
- **优势**: 全双工通信延时最低。
- **劣势**: WebSockets 在弱网环境断图需要重做握手和心跳处理，管理 ASGI 下的连接池也相对复杂，容易在代理层发生阻塞断流（例如 Caddy 代理需特殊优化）。

### 方案 C (SSE 流推送)
- **优势**: 单向推拉模式。前端请求仍然是完全标准的 HTTP 协议，不存在握手心跳黑箱，并且对 Caddy 的要求极低（只需 `X-Accel-Buffering: no`）。同时复用了我们在方案 **0006 架构演进** 中的 Redis 通道（仅仅多声明一个 `board:events` channel）。符合项目原有的强行状态机哲学。
- **劣势**: 前端发送内容时依然是传统的 AJAX，稍微增加了一丁点 HTTP 包头开销。但在此场景（非微秒级抢答交易）完全忽略不计。

## 4. 最终决定
采用 **方案 C**，彻底禁止引入复杂的 WebSockets。通过原生 Python 的 `SQLAlchemy` 模型加上 RLS 机制控制读写权限，结合单向的 `SSE 流` 做消息到达的轻推送，在 Vue 侧合并为平滑动画。保证了 0 新增中间件成本！

## 5. 影响范围
- **依赖库精简**：完全无需再安装 WebSocket 的复杂库。
- **兼容性**：移动端和受设备高度约束的极简浏览器中也有着极为优秀的兼容表现（只需依赖 `EventSource`）。
