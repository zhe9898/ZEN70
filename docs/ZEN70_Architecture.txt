# ZEN70 架构实施法典 (V1.58 终极满血版)

## 第 1 部分：基础设施、全局网络与拓扑状态机 (V1.58 终极封版)

### 1. 全栈成熟技术路线规约 (Technology Stack Specification)

为确保系统在长生命周期内的可维护性与平滑演进，全栈技术选型强制采用以下工业级成熟方案。系统必须实现软件逻辑与物理硬件的彻底解耦。所有硬件资源均抽象为“能力（Capabilities）”标签，严禁在代码与配置中硬编码特定的主板、CPU、GPU、硬盘型号或单一商业软件栈，全面支持跨指令集（x86/ARM）与跨平台的演进。

【技术实施细节与路线说明】
• 能力标签规范化 (Capability Registry)：在全局配置中，硬件能力标签必须遵循标准化的语义命名与版本控制（如 gpu_nvenc_v1）。不同驱动版本的能力差异必须在标签层面体现。
• 运维配套 (SOP)：由于架构涉及多级熔断与复杂防御逻辑，必须配套编写标准化的《运维手册与故障演练剧本 (Chaos Playbook)》，将架构知识转化为一线运维人员的标准操作程序。

#### 1.1 基础设施与容器编排层 (Infrastructure & Orchestration)
• 容器编排：强制采用 Docker Daemon + docker-compose v3.8+（声明式编排）。所有微服务必须实现彻底的容器化，宿主机仅作为提供基础计算与存储资源的“无状态节点”。
• 供应链安全体系：采用自建私有镜像仓库（Harbor 或轻量级 Registry）与 Git 多源同步机制。核心服务的镜像及配置数据必须多点备份。
• 自动化初始化部署：使用跨平台 Python 脚本执行自动化部署。脚本需内嵌多源重试逻辑，负责环境基线预检、配置拉取和容器启动，支持在外部代码托管平台（如 GitHub）不可用时自动切换至备用源（如 Gitee/自建服务），保障离线环境下的部署能力。

【技术实施细节与路线说明】
• Docker API 交互：初始化脚本（bootstrap.py）应使用官方 docker Python SDK 以准确捕获 Daemon 状态与容器异常。
• 镜像加速与重试：配置拉取函数必须包含 try...except 块，首选 GitHub Raw 链接，失败后自动切换至备用镜像源。私有 Registry 需配置 TLS 证书及 Basic Auth，确保节点拉取镜像的安全闭环。
• IaC 极简克制原则：点火脚本仅负责最基础的环境预检与容器拉起。复杂的模板渲染与业务逻辑统统交由专用的配置编译器处理。编译器严格遵循“Jinja2 模板零复杂业务逻辑（仅做变量替换）”原则，部署前必须强制运行 config-lint 校验配置。

#### 1.2 全局网络与路由层 (Network & Routing)
• 公网访问隧道：使用 Cloudflare Tunnel 提供泛域名 HTTPS 直连。规避运营商端口限制及流量劫持，为普通用户提供低门槛的公网访问能力。
• 高可用私有虚拟网：采用 Headscale + WireGuard 构建基于 UDP 打洞的 P2P 虚拟局域网。鉴权密钥等核心数据必须挂载至持久化数据卷，并配置容器自动重启策略（restart: unless-stopped），保障极端网络环境下的节点互联。
• 反向代理网关：统一使用 Caddy（支持自动化 HTTPS 证书流转）或 Nginx，负责南北向网络流量的转发与请求体控制。

【技术实施细节与路线说明】
• 网络平面隔离：在 docker-compose.yml 中，强制定义两个网络平面：frontend_net（供反向代理与隧道通信）和 backend_net（供后端微服务间内部通信）。
• Caddy 配置规范：推荐使用 Caddy 替代 Nginx，配置文件 Caddyfile 需设定严格的 reverse_proxy 路由，并开启 encode gzip zstd 压缩。

#### 1.3 后端核心服务与状态机 (Backend API Gateway)
• API 核心网关：采用 Python 3.11+ 配合 FastAPI 框架（强制要求 Pydantic v2）。基于原生 asyncio 实现异步非阻塞 I/O。
• 状态同步机制：基于 Server-Sent Events (SSE) 实现后端到前端的低延迟状态单向推送（用于硬件拓扑变化与开关状态广播）。后端需配备客户端心跳检测，若超时未收到心跳则自动清理僵尸连接。
• 拓扑监测探针 (Topology Sentinel)：轻量级 Python 单文件守护进程，仅依赖内置库与 Redis。
• 职责边界：核心负责硬件热插拔检测（挂载点/GPU）、触发状态更新策略，以及向 Redis 写入短期防抖锁。
• 功能克制：严禁在探针中集成复杂的业务时序指标采集。周期性资源监控统一交由 Categraf 负责。支持状态锁的心跳续期，探针崩溃后锁自动过期。
• 【强制熔断顺序】：探针与网关联动，检测到硬件缺失或微服务心跳异常后，系统必须严格按以下顺序执行降级，严禁直接操作容器：
1. API 熔断与就绪探针 (Readiness Probe)：网关接收到状态锁，或连续 3 轮探测微服务内部 `/healthz` 失败后，立即对该路由返回 HTTP 503，拒绝新请求进入，阻断流量。
2. 网络断流：在 MVP 阶段，依赖第一步的网关本地熔断。通过调用 Consul deregister API 动态更新 upstream 的物理断流动作逐步引入。
3. 容器降级与存活探针 (Liveness Probe)：对于纯本地 I/O 读写型容器执行声明式状态变更（自动调谐）；对于死锁（进程存活但 HTTP 超时）的容器，触发存活探针的强杀重启，确保依赖链断开。

【技术实施细节与路线说明】
• 网关本地副本收敛协议：FastAPI 网关启动时必须从 Redis 加载全量状态，并在进程内维护一份只读的 LRU 缓存（TTL ≤ 30 秒），缓存内容明确为“硬件状态字典”与“全域能力矩阵”。
• 一致性窗口妥协声明：当 Redis 发生单点故障失联，网关立刻转入“全域悲观静默”模式。运维人员必须明确：在 Redis 宕机期间，硬件的真实物理跳变无法被系统感知，此为系统在“一致性 vs 可用性”中的刻意妥协。一切状态必须待 Redis 恢复后，网关清空本地脏缓存并重拉全量状态后方可生效。
• 边缘节点缓存配额管治：在集群扩展场景中，边缘节点与主控失联时，仅允许执行“停止从队列拉取新任务（Stop-Pulling）”。已在执行中的任务允许在本地内存中继续运行并缓存结果。强制限制：缓存结果必须设定上限（如磁盘配额 ≤ 500MB，或最大记录条数），且设置 TTL（如 24 小时后自动丢弃），防止长时间失联撑爆边缘节点磁盘。

#### 1.4 数据持久化与缓存层 (Database & Cache)
• 关系型数据库与向量引擎：采用企业级 PostgreSQL 15+ 配合 pgvector 插件。承担系统核心元数据、功能开关状态以及 AI 会话 Embedding 特征的存储。
• 高速缓存与分布式锁：采用 Redis 7+（强制开启 AOF 持久化）。专用于高频的硬件在线/离线状态缓存、API 接口限流、分布式锁管理以及 SSE 发布/订阅消息总线。

【技术实施细节与路线说明】
• PostgreSQL 向量调优：系统默认采用构建速度快、内存开销小的 IVFFlat 索引。严禁以硬编码的静态数据量阈值决定索引类型。运维团队应根据监控反馈的查询模式与硬件内存余量，参考提供的调优指南，动态决定是否手动切换为 HNSW 索引。
• Redis AOF 策略：Redis 的 redis.conf 必须显式设定 appendonly yes 及 appendfsync everysec。

#### 1.5 前端交互层 (Frontend UI/UX)
• 核心渲染框架：采用 Vue 3 (Composition API) + Vite 构建工具 + TypeScript。
• 跨端应用载体：严格遵循 PWA (Progressive Web App) 规范。利用 manifest.json 与 Service Worker，支持用户将网页添加至设备桌面，以独立沙盒进程运行。
• 离线存储机制：利用 Dexie.js 深度封装浏览器底层 IndexedDB。用于缓存基础 UI 骨架、核心状态数据及静态资源。
• 无密码安全鉴权：集成 WebAuthn (FIDO2) 协议。调用设备底层的安全芯片实现跨设备的无密码生物识别登录。

【技术实施细节与路线说明】
• PIN 码指纹降级管控：针对局域网环境及生物模块故障场景提供的 6 位 PIN 码降级登录方案，必须实装严格的指数退避（首次失败等待 1 秒，二次 5 秒，三次 30 秒）。
• 锁定维度区分：连续失败 5 次后，强制锁定该请求源 15 分钟。锁定机制必须以 请求 IP 为主要防御维度。设备指纹（如 IndexedDB 存储的 clientId 或 IP + User-Agent 组合哈希）仅作为辅助因子，以防止 NAT 环境下的合法用户误伤。系统必须提供管理员手动解锁接口，作为误锁时的终极逃生通道。

## 第 2 部分：后端核心总线、AI 多模态联邦与协议驱动前端 (V1.58 终极满血版)

### 2.1 全栈成熟技术路线规约 (Technology Stack Specification)
为确保系统在长周期内的可维护性与平滑升级，前后端分离架构的技术路线必须严格遵守以下规范。本模块已实现系统逻辑与硬件底层的绝对解耦，所有硬件均以“能力（Capabilities）”进行抽象。

#### 2.1.1 后端中枢服务 (Core API Gateway)
• 核心框架：强制采用 Python 3.11+ 配合 FastAPI（要求 Pydantic v2）。利用原生 asyncio 实现异步非阻塞 I/O，最大化任意异构算力节点的并发性能。
• 状态同步与连接高可用：基于 Server-Sent Events (SSE) 实现后端到前端的单向状态推送。强制实装心跳探活机制：前端客户端需每 30 秒发送 Ping 探活请求；后端若连续 45 秒未收到 Ping 信号，必须主动调用 cancel() 强行斩断该 SSE 挂起协程，释放文件描述符（FD）与内存，杜绝半开连接导致的资源耗尽。
• [实装 建议7] 统一错误码契约与自愈指导：废除模糊的 500 错误。定义全局错误码字典，强制引入 recovery_hint（恢复建议）字段。前端可据此动态渲染操作按钮，指导用户自愈。

【技术实施细节与路线说明】
• 网关本地副本的一致性妥协 (CAP 权衡)：FastAPI 网关内部必须维护一份 LRU 缓存（TTL ≤ 30 秒），缓存内容明确为**“硬件状态字典”与“全域能力矩阵”**。在 Redis 故障期间，系统牺牲强一致性保全高可用性。Redis 宕机期间硬件的真实物理跳变无法被系统感知，网关转入“悲观静默”。Redis 恢复后，网关必须立即清空本地缓存并重新拉取全量状态，严禁网关向 Redis 反向写回状态。

#### 2.1.2 前端交互与展现层 (Frontend UI/UX)
• 核心渲染引擎：采用 Vue 3 (Composition API) + Vite，确保在性能受限设备上的极速渲染。
• 应用形态 (PWA)：严格遵守渐进式 Web 应用规范。通过 manifest.json 与 Service Worker，实现网页脱离浏览器外壳独立沙盒运行，规避原生 App 的审核与维护成本。
• 无密码鉴权与离线零信任令牌：
• 在线状态强制绑定 WebAuthn (FIDO2) 协议，调用设备安全芯片（Face ID / 指纹）实现无密登录。
• 离线降级沙盒：用户首次联网登录后，系统自动签发一份低权限、仅限读取的离线专用 JWT，并利用 Web Crypto API 加密存入 IndexedDB。断网时跳过 WebAuthn 挑战应答，直接解密该令牌进入“只读骨架屏模式”，保障断网环境下的基本可用性。

【技术实施细节与路线说明】
• PIN 码降级与指纹弱依赖：为防 WebAuthn 失效，系统提供局域网 IP 限定的 6 位 PIN 码降级方案。该接口必须实装指数退避（失败 1次 等待 1s，2次 5s，3次 30s）。连续失败 5 次后，强制锁定 15 分钟。锁定机制必须以请求 IP 为核心防御维度，设备指纹（通过 IndexedDB 存储的 clientId 或 IP + User-Agent 哈希）仅作为防止 NAT 误伤的辅助因子。系统必须提供管理员手动解锁接口。

#### 2.1.3 数据持久化与演进库 (Database)
• 核心记忆总线：采用企业级 PostgreSQL 15+ 配合 pgvector 向量引擎。存储系统元数据及 AI 会话 Embedding 特征。
• 高速状态锁：采用 Redis 7+（开启 AOF 持久化），专用于高频拓扑状态缓存、API 限流及分布式锁管理。

【技术实施细节与路线说明】
• 向量索引动态调优：系统初始化时，默认使用构建速度快、内存开销小的 IVFFlat 索引。严禁以静态的“10万条”作为硬编码切换阈值。必须通过 Categraf 采集慢查询日志（Slow Query Log）及内存利用率，在运维 SOP 中提供明确的调优边界，由运维人员动态决定是否重建为 HNSW 索引。

#### 2.1.4 拓扑监测探针 (Topology Sentinel)
• 极简技术栈：仅依赖 Python 内置库与 Redis，单文件守护进程。
• 探针看门狗与死锁防范：探针负责检测挂载点存活并向 Redis 写入状态锁。对于已确认的离线状态锁（如 PENDING_MAINTENANCE），探针必须每隔 5 秒发送心跳续签。若探针进程崩溃导致心跳中断超过 15 秒，Redis 锁自动消散，交由网关层接管，彻底避免探针单点故障导致的系统永久锁死。

【技术实施细节与路线说明】
• 边缘节点静默待命原则：在横向集群扩展场景中，边缘工蜂节点一旦发现无法连接主控 Redis，仅允许执行一个动作：停止从队列拉取新任务（Stop-Pulling）。严禁触发主动杀容器（pause/stop）的自裁决。
• [实装 建议5] 边缘滞留任务的配额管治与状态联动：已在执行中的任务允许在边缘节点本地内存中继续运行并缓存结果。强制限制 1：缓存必须设定绝对上限（如磁盘配额 ≤ 500MB，TTL 24 小时）。强制限制 2：边缘节点在网络恢复后上报缓存结果时，必须附带 Task ID 与时间戳。主控节点负责执行校验，若该任务在此期间已被标记为超时或取消，主节点将直接丢弃结果并指令边缘节点清理对应缓存，避免逻辑错乱与算力浪费。

### 2.2 AI 联邦与多模态 Map-Reduce 调度总线
针对复杂的多模态输入（语音、视觉、文本），架构引入分布式调度与异步熔断机制，防止异构节点的显存溢出（OOM）。

#### 2.2.1 并发解包与路由 (Parser & Routing)
网关接收多模态载荷后进行流切分：音频流路由至主控 CPU 线程（调用轻量级语音模型）；图像流并行分发至边缘异构张量计算节点（调用视觉大模型）。

#### 2.2.2 异步并发与极刑防死锁机制
严格设定动态熔断超时阈值。开发人员必须遵守异步超时规约（asyncio.timeout）。一旦捕获超时（TimeoutError），必须显式下发取消指令，并向前端返回纯文本的兜底响应，确保核心服务不被拖垮。

【技术实施细节与路线说明】
• [实装 建议2] 【底层资源强制释放红线】：鉴于 Python asyncio.cancel() 取消机制的协作式局限性（无法中断底层正处于阻塞状态的 CUDA 调用），针对重型 GPU 任务，严禁单纯依赖协程取消。必须采用子进程池隔离运行（multiprocessing）并配合 terminate() 强制终结系统进程；或要求底层 AI 推理接口（如 Triton/vLLM）原生提供并支持强超时的 API 参数，从底层内核级别保障异构加速卡显存句柄的绝对释放。

#### 2.2.3 动态调度算法
废除静态轮询。基于节点实时负载（CPU/GPU 利用率、队列长度）动态调度任务，自动将推理请求分发至最空闲节点。

【技术实施细节与路线说明】
• [实装 建议1] 能力标签标准化与版本感知 (Capability Registry)：调度器依赖的能力标签必须遵循标准化的命名约束与版本感知。标签版本（如 gpu_nvenc_v1）由节点探针探测底层驱动（如 nvidia-smi 输出）后自动生成并上报。调度器仅将任务分发给完全匹配该标签版本的节点。系统升级时需同步更新节点驱动与标签字典表，确保任务分发的后向兼容性。

### 2.3 硬件拓扑自适应与“协议驱动 UI”开关矩阵
实行全域动态协议渲染（Schema-Driven UI），废除前端硬编码控制按钮。

#### 2.3.1 后端协议动态生成
结合底层的硬件存活状态，API 网关动态生成 JSON Schema 协议。若探针发现存储阵列离线，对应的模块在能力矩阵中将被标记为 disabled。前端通过 /api/v1/capabilities 获取矩阵并自动利用 v-for 渲染控制卡片。

#### 2.3.2 软硬联动大闸与极刑强杀
• 硬件缺失响应：外部硬件离线时，软开关自动变为不可交互的灰色状态。
• 硬件接入恢复：新硬件插入被识别后，控制开关解除锁定。点击开启时，强制使用 docker restart 替代 unpause，以此强制刷新 Linux 底层文件系统的 Inode 映射，彻底解决旧挂载句柄失效导致的 I/O 读写错误。

【技术实施细节与路线说明】
• 【强制熔断顺序约束】：在应对硬件缺失时，系统必须严格按以下顺序执行：
1. API 熔断：网关立即返回 503，拒绝新请求进入。
2. 网络断流：从内部服务发现中摘除故障路由。
3. 容器降级：向底层下发降级指令。对于纯 I/O 容器执行 docker pause（必须包裹 3 秒硬超时，超时升级为 SIGKILL 强杀）；对于有复杂网络依赖的有状态容器，强制执行 docker stop -t 10 优雅下线。
• [实装 建议4] 网络断流的 MVP 局限性与演进声明：在初始 MVP 阶段，允许通过 API 网关的本地内存退避拦截来实现极简版断流。但必须在工程文档中明确其局限性（下游依赖方可能因疯狂重试导致资源耗尽）。真正的动态服务发现摘除（如调用 Consul API 或动态 reload Nginx Upstream）必须在架构稳定后作为 P0 级需求引入。

#### 2.3.3 降级预案：悲观静默
当网关内存副本过期或验证到底层真实 I/O 阻断时，直接转为“悲观静默（Pessimistic Drop）”，返回 503 拒绝新请求，保护底层并发池。

### 2.4 跨平台使用场景适配
针对割裂的移动生态，提供深度的兼容性保障：
• 定制安卓系统：通过加密 HTTPS 隧道直连并添加 PWA 至桌面，规避流氓广告劫持与激进的后台进程查杀。
• 苹果 iOS 生态：深度集成 Safari 的 PWA 与原生 HLS 播放能力。调用设备硬件解码器，并支持 Face ID 无感鉴权。
• 原生安卓生态：兼容完整的 Google 移动服务（GMS），接入 Media Session API 与 Web Push。

【技术实施细节与路线说明】
• [实装 建议3] Service Worker 离线缓存优化策略：静态核心资源采用 CacheFirst；状态 API 采用 NetworkFirst。针对视频流切片等重型媒体数据，严禁采用暴力的 NetworkOnly 策略（会引发重复播放时的带宽严重浪费），必须采用带有严格存储上限（如 maxSize: 200MB）结合 LRU（最近最少使用）淘汰机制的 StaleWhileRevalidate 或动态缓存策略，兼顾带宽节省与设备存储安全。

### 2.5 架构的平滑演进与优雅降级评估
• 情绪隔离与优雅降级：当重型硬件掉线时，系统拦截前端报错。前端利用 IndexedDB 提取缓存的海报并覆盖磨砂滤镜，提示“设备维护中”，实现物理故障与用户体验的隔离。
• 无极横向扩展：系统预留标准 RESTful 边界，通过协议动态渲染新接入的模块交互界面。

【技术实施细节与路线说明】
• [实装 建议6] 基于角色的错误暴漏机制 (RBAC UI)：前端骨架屏的“情绪隔离”绝不能成为阻碍极客排障的黑盒。系统必须实施基于角色的错误暴露机制。普通家庭账户渲染温和的磨砂提示（“设备维护中”）；而在具有“极客/指挥官”权限的账户下，允许在组件折叠面板中直接展示包含 TraceID 与后端原始状态码（如 502/503）的追踪日志，保障极速排障能力。
• 异常排查知识传递：开发团队必须配套编写故障演练剧本（SOP）。指导运维人员通过 Grafana 等监控指标快速判定降级大闸触发的根源。

## 第 3 部分：数据持久化、容灾备份与全域安全边界 (V1.58 终极满血版)

### 3.1 全栈成熟技术路线规约 (Technology Stack Specification)
为确保数据层的绝对安全与平滑演进，数据持久化与灾备管线必须严格遵守以下技术栈规范。系统已实现存储介质与算力芯片的彻底抽象化，严禁在配置中硬编码任何物理硬盘容量或主板型号。

#### 3.1.1 关系型与高维特征数据库 (Primary DB)
• 核心数据库：采用企业级 PostgreSQL 15+。承担系统核心元数据、功能开关状态、用户配置及审计日志等结构化数据存储。
• 向量化扩展：启用 pgvector 插件，用于存储 AI 会话的 Embedding 特征，支撑基于 RAG（检索增强生成）的系统演进。
• 索引调优：系统初始化时，默认采用构建速度快、内存开销小的 IVFFlat 索引。严禁以静态数据量作为硬编码切换阈值，必须在环境变量中暴露索引类型（HNSW / IVFFlat）及 lists 参数，结合实际监控反馈进行动态调优。
• 架构红线：严禁使用轻量级本地文件库（如 SQLite）作为核心状态并发总线，以防高并发场景下的数据库死锁或写阻塞。

#### 3.1.2 高速状态缓存与限流器 (Cache & Lock)
• 缓存选型：采用内存级高速键值对数据库 Redis 7+。
• 核心职责：管理高频的硬件在线/离线状态缓存、API 接口限流（Rate Limiting）、SSE 长连接的发布/订阅背板以及分布式锁。
• 高可用配置：强制开启 AOF 持久化，并配置 appendfsync everysec，确保异常重启后状态能实现秒级恢复。

#### 3.1.3 容灾备份引擎 (Disaster Recovery)
• 备份工具：采用支持块级去重与 AES-256 强加密的备份工具 Restic。
• 存储后端：配合 S3 标准协议的云端对象存储（如 Cloudflare R2、AWS S3），实现异地灾备。
• 不可变存储 (Anti-Ransomware)：强制开启 S3 对象存储的 Object Lock（合规保留期）功能。确保备份数据在设定周期内处于 WORM（Write Once Read Many）状态，防御勒索病毒及凭证泄露带来的删库风险。
• 数据分级标签 (FinOps 控制)：在 system.yaml 中为挂载点引入 backup_tier 字段：
• critical：核心配置与数据库，强制推流至云端 S3。
• ephemeral：临时缓存，跳过备份。
• media：大容量媒体冷数据，仅执行本地异盘备份。

#### 3.1.4 零信任安全鉴权边界 (Zero-Trust Auth)
• 认证协议：执行 WebAuthn (FIDO2) 无密码标准。结合设备安全芯片（Face ID/指纹/TPM）完成活体核验，签发短期 JWT 令牌。
• 自动化令牌 (PAT) 面向机对机控制：支持生成带有生命周期的 Personal Access Token，配合 Webhook 使用。
• 双轨令牌轮转与防越权：PAT 必须支持独立吊销，且在 Redis 中维护黑名单机制。实现长效令牌轮换期间的新旧并行过渡。
• 紧急恢复：提供基于 Shamir 秘密共享机制分割保管的高强度一次性恢复码，防止单点设备丢失导致系统失联。

【技术实施细节与路线说明】
• Restic 不可变存储配置：初始化 Restic 仓库时，后端自动化脚本必须带上 --repository-version 2 参数，以支持 S3 Object Lock。备份写入后，系统需通过 S3 API 显式配置对象的 Retain Until Date（如 30 天），确保管理员权限也无法在期限内删除。
• Shamir 密钥分片逻辑：初始化脚本调用 Python SecretSharing 库，将最高权限的 Master Key 拆分为 5 份（n=5），配置恢复门限为 3 份（k=3）。前端生成可供打印的二维码与纸质密钥，要求用户分发至不同物理地点。

### 3.2 硬件拓扑自适应感应与数据 I/O 保护 (Hardware Sensing & I/O Protection)
针对物理存储意外断开或数据静默损坏，系统通过“探针-网关-容器”联动机制实施自愈防御。

#### 3.2.1 应对硬件缺失的降级熔断机制
• 故障侦测：Topology Sentinel（拓扑探针）以 5 秒周期轮询物理挂载点状态。
• 悲观锁拦截：探针发现路径丢失后，在 15 秒的防抖确认期内，向 Redis 写入 PENDING_MAINTENANCE 短期锁（TTL 20 秒）。API 网关拦截所有涉及该硬件的请求，防止无效 I/O 撑爆系统盘。
• 容器异步冻结：确认为物理掉线后，网关向容器引擎下发降级指令。操作必须异步执行并设定 3 秒硬超时，若因进程陷入不可中断睡眠（D 状态）导致命令挂起，自动升级为 SIGKILL 强行斩断内核句柄。
• 状态同步：状态机更新为 OFFLINE，通过 SSE 广播前端，对应功能开关置灰锁定。

#### 3.2.2 应对硬件接入的恢复机制
• 三重交叉核验：新存储介质接入后，探针必须通过三项验证才允许解锁：1. 路径存在性 (os.path.exists)；2. UUID 一致性校验（比对 system.yaml 声明）；3. 最小剩余容量检测（防止空间耗尽错误）。
• 容器重载：验证通过后，前端控制台自动解锁。用户点击开启时，系统执行 docker restart（废除存在 Inode 映射风险的 unpause），彻底刷新底层文件系统描述符，恢复相关服务。

#### 3.2.3 静默数据腐败巡检 (Bit-Rot Detection)
• 通过低优先级后台任务，在核心 CPU 负载低于 10% 时，对底层冷数据块进行 SHA-256 哈希比对。若发现物理扇区数据反转，即刻通过告警通道推送预警以实施数据隔离。

#### 3.2.4 UPS 电源联动与优雅停机 (Graceful Shutdown)
为防止市电突发中断导致数据库撕裂与文件系统损坏，正式引入网络 UPS 工具（NUT）联动机制：
• 电源状态监听：部署轻量级 nut-client 守护容器，通过 USB 或局域网持续监听 UPS 供电状态与电池余量。
• 全局只读熔断：当监测到市电断开且 UPS 电池余量低于 20%（或预估运行时间小于 5 分钟）时，触发最高级别中断。API 网关立刻拒绝所有 POST/PUT/DELETE 写操作，系统进入全局只读模式。
• 数据落盘与停机：系统自动触发 Redis 的 BGSAVE 与 PostgreSQL 的 CHECKPOINT 指令强制内存数据落盘。随后依次下发 docker stop 优雅终止所有业务容器。确认核心容器退出后，调用宿主机 halt 指令完成安全关机。

【技术实施细节与路线说明】
• 【强制 I/O 熔断顺序约束与 HTTP 语义规范】：在 3.2.1 的硬件缺失响应中，代码实现必须严格遵循：① API 熔断（针对同步强读写请求绝对阻断并抛出 503 Service Unavailable；针对非核心异步排队任务，挂起至等待队列并返回 202 Accepted）；② 网络断流（从网关层摘除对该容器的 Proxy 转发）；③ 执行降级指令。
• 静默数据腐败哈希基线库 (规避 3.1.1 冲突)：Bit-Rot 巡检 Worker 在首次扫描时，需将大容量冷数据的哈希基线存入本地验证库。此处明确允许使用独立的 SQLite 文件。由于该库与核心业务完全隔离，仅供后台 Worker 进行极低频、单线程的本地读写校验，完美规避了 3.1.1 节关于“严禁使用 SQLite 作为高并发核心总线”的安全红线。
• 优雅停机阻塞等待：在 UPS 触发停机脚本时，发送 CHECKPOINT 给 PostgreSQL 后，必须包含一个 while 轮询检查 pg_stat_activity 确保刷盘完毕。调用 docker stop 时，需追加 --time 30 参数，给予数据库容器足够的内存收尾时间，之后再执行 halt。

### 3.3 跨端使用场景的优雅降级与数据合规 (Cross-Platform UX & Data Governance)
架构深度考量了离线故障期间的跨平台体验连续性及数据隐私合规性。

#### 3.3.1 生态级 UI 降级保护
• 异常隔离：当存储阵列掉线时，前端应用拦截后端下发的 Storage_Engine: OFF 协议，杜绝显示代码级报错（如 502/404）。
• 骨架屏渲染：利用 Dexie.js 从 IndexedDB 提取缓存的界面数据（如海报/快照），置灰并叠加磨砂滤镜，提示“设备正在维护”，实现物理灾难与用户体验的工程隔离。

#### 3.3.2 数据生命周期终局管控
• 数据可携带权 (Data Portability)：支持按用户维度一键全量导出关联资产及日志（JSON/CSV 格式），满足合规审查要求。
• 安全覆写销毁 (Secure Shredding)：通过二次生物鉴权后，支持不仅删除数据库记录，同时调用底层命令对物理磁盘区块进行多次无规则碎片化覆写，确保数据不可被物理恢复。

【技术实施细节与路线说明】
• 安全覆写算法 (Shredding API)：对于触发安全销毁的挂载点文件，后端 Python 脚本不得仅调用 os.remove()。必须通过 subprocess 调用宿主机的 Linux shred 指令（如 shred -u -z -n 3 <filepath>），执行 3 次随机数据覆写并以零填充结尾，从底层物理磁道抹除数据残留。

### 3.4 容灾备份的闭环验证 (Disaster Recovery Validation)
备份系统必须具备防干扰与自校验能力，杜绝“薛定谔的备份”。

#### 3.4.1 负载感知避让
• 在执行增量加密推流前，脚本主动核对 CPU 与异构加速卡的实时负载。若系统正处于高并发推理或重型流媒体转码状态，备份任务自动挂起并推迟，保障前台核心业务的 SLA。

#### 3.4.2 备份完整性抽检
• 推流完成后，强制执行 restic check --read-data-subset，随机下载 5% 的远端数据块进行解密与校验。仅当校验 100% 通过时，更新数据库中的备份健康度状态；失败则自动触发告警重试。

#### 3.4.3 异构环境恢复
• 系统恢复过程不依赖原有的 CPU 指令集架构或主板环境。在安装标准容器引擎的新设备上，导入密钥与点火脚本即可从对象存储拉取快照，快速重建包含状态机在内的全量系统。

【技术实施细节与路线说明】
• 负载指标获取逻辑：备份脚本（由 Cron 触发）在启动前，直接查询本地 Prometheus/Categraf 暴露的 /metrics 端点。设定阈值：当 cpu_usage_active > 75% 或 gpu_utilization > 80% 持续超过 3 分钟时，脚本以 Exit Code 0 静默退出，并在日志中记录 Skipped due to high load，交由下一次 Cron 周期处理。
• 离线点火与 S3 交互安全边界 (IaC 握手)：在“第 0 天”灾难恢复（裸机无任何存量配置）时，跨平台点火脚本（bootstrap.py）必须通过人机交互安全终端（如调用 Python getpass 模块隐藏输入）或挂载临时解密物理盘的方式，向系统安全注入临时的 S3 AK/SK 环境变量。脚本获取短期授权后，优先调用 Restic 直接从云端拉取最新的 system.yaml 与全局环境密文，随后立即销毁内存中的 AK/SK 变量，彻底阻断凭证泄露途径。

### 3.5 物联指令的防御性控制 (IoT Defensive Control)
确保系统接管智能家居协议层时的容错率与边界安全：
• 指令幂等性与去重：每一个下发的物联网控制指令必须携带 UUID（Command ID）。执行 Worker (Consumer) 在消费时，必须根据该特征码对指令进行全局去重拦截，防止重复触发导致的设备紊乱。
• 死信队列与衰减重试 (DLQ)：对于离线设备的异步控制，需配置指数退避机制。重试计数超过最高阈值（如三次）的僵尸消息，必须排入死信队列 (Dead Letter Queue)，并立即触发告警，绝对禁止无限轮询挤占 Redis 带宽。
• 内部通信 mTLS 强制化：Redis Pub/Sub 频道和 MQTT Broker 的内网通讯必须启用 TLS 加解密证书。彻底抹杀局域网链路层的明文嗅探风险。

### 3.6 声明式闭环与微服务容灾调度 (K3s-Inspired SRE)
为了摒弃单线程的“脚本指令式”脆弱执行（如单纯的 `if offline then pause`），ZEN70 架构汲取大规模集群调度思想（K3s），在单节点或边缘集群中全面落地「声明式控制循环」与柔性保护大闸：

#### 3.6.1 污点与容忍度 (Taints & Tolerations)
核心功能不再非黑即白地直接熔断。当探针检测到底层严重负面状态（如温度 > 85°C，或 UPS 电池续航告急）时，会向系统总线写入「污点 (Taint)」（如 `status=overheating:NoSchedule`）。
• **容忍度 (Toleration)**：基础监控容器与 API 网关自带该污点的容忍度，继续坚守岗位；而重负载如媒体转码、AI 索引等后台 Worker，将主动避让该污点立刻挂起，实现系统的平滑「柔性降级」。

#### 3.6.2 声明式控制循环 (Reconciliation Loop)
废除触发器式脚本，将引擎核心重写为永不退出的「闭环控制 (Reconciliation Loop)」。
• 引擎只执行三大动作：**观察 (Observe) -> 找差异 (Diff) -> 执行对齐 (Act)**。
• 系统持续比对 `system.yaml` 中期望加载的配置字典，与底层容器 Socket 反馈的实际存活清单。一旦发现偏离（例如关键服务被误杀），无需编写异常判断代码，引擎自行像弹簧一样利用 `docker-compose up` 将状态强制弹回对齐位置。

#### 3.6.3 深层健康探测 (Liveness & Readiness Probes)
舍弃通过 `docker inspect` 判断容器表象是否 running 的漏洞做法。API 网关与探针必须直接发起 HTTP GET 核验内部真实业务状态。
• **就绪探针 (Readiness)**：大模型容器刚启动，模型尚未 Load 完毕时，网关切断该分支流量，确保进入网关的请求得到 100% 接待。
• **存活探针 (Liveness)**：模型死锁（进程未挂但卡死），连续三次 `/healthz` HTTP 超时，探针直接越过 Docker 下发僵尸清场重启大闸。

#### 3.6.4 边缘孤岛的墓碑驱逐机制 (Eviction & Tombstones)
针对多节点或弱网环境，边缘节点失联超 15 秒（3 次心跳丢失）即被逻辑判定为「死亡驱逐」。
• **墓碑 (Tombstone)**：主控在 Redis 放下对应任务的墓碑，并重新调度该任务至健康节点计算。
• **恢复对齐**：当断网边缘节点重新连线后，其首要任务是查阅墓碑列表。凡命中其本地排队中的「已死亡任务」，该节点必须自行清空本地缓存将其强制丢弃，彻底杜绝数据双写与算力空耗的“僵尸复活”灾难。

## 第 4 部分：边缘娱乐引擎、家庭关怀视图与全域物联总线 (V1.58 终极满血版)

### 4.1 娱乐与物联技术路线规约
为确保流媒体分发与物联体系的长期可维护性，本模块实现算力与存储介质的彻底抽象化，严禁硬编码具体的处理器、加速卡或阵列型号。所有组件必须基于工业级开源项目构建。

#### 4.1.1 核心流媒体引擎
• 引擎选型：采用纯净版、容器化的开源流媒体服务（如 Jellyfin）。
• 架构要求：必须支持全量 API 深度定制，满足前后端分离及无头（Headless）调用需求。配置统一通过环境变量注入，通过标准卷挂载访问底层存储阵列。

#### 4.1.2 硬件编解码管线
• 底层框架：强绑定 FFmpeg 框架。通过容器运行时工具（如 NVIDIA Container Toolkit）直通异构加速卡的专用硬件编解码单元（如 NVENC/NVDEC、Intel QSV、Apple VideoToolbox）。
• 能力抽象：编解码能力由探针检测并以标签形式上报（如 ["h264_enc", "hevc_dec"]）。调度器根据任务需求动态分配资源，硬件更替无需修改核心代码。

#### 4.1.3 本地视觉计算引擎 (NVR)
• 引擎选型：采用基于本地张量计算的实时 AI 监控引擎（如 Frigate）。
• 技术要求：利用目标检测模型实现精准识别（人、车、动物），摒弃基于像素变化的传统移动侦测机制。原生支持通过 MQTT 协议与物联总线联动。
• 隐私管控：默认禁用面部特征等生物识别功能，启用需配置独立的隐私合规策略。

#### 4.1.4 全域物联消息总线
• 总线选型：部署 Eclipse Mosquitto (MQTT) 作为轻量级发布/订阅中枢。
• 核心职责：负责安防事件、环境传感器与核心网关间的低延迟通信。强制配置宿主机目录持久化及 restart: unless-stopped 策略，确保服务高可用。

#### 4.1.5 前端播放器组件
• 采用 hls.js 与 video.js 封装的 Vue 3 组件，兼容各平台底层硬件解码标准。支持自适应网络码率切换，并在流媒体服务不可达时，自动降级展示本地缓存的界面骨架（缓存机制遵循第 6 部分离线存储规范）。

#### 4.1.6 物联指令控制总线 (IoT Command Control Bus)
• 中继层健康感知：引入 HomeAssistant 或 Zigbee2MQTT 作为隔离翻译层，网关需通过异步节点轮询获取中继层的心跳。若中继离线，网关需立即向前端反馈 503 HTTP 状态并将本地指令缓存在 Redis，以备上线重发，而不是陷入 TCP 握手阻塞的死等待。
• SSE 状态物理防抖确认：当且仅当翻译网关通过 MQTT 送回设备的真实回执时，前端看板才能通过 SSE 接收到开关已响应的事件。杜绝因硬件掉线造成的伪状态指示。

【技术实施细节与路线说明】
• 挂载权限红线 (Read-Only Mounts)：流媒体容器（如 Jellyfin）对底层大容量物理阵列的映射必须强制声明为 ro（只读）。元数据、转码缓存与配置文件必须挂载至宿主机的独立 SSD 目录（rw 权限）。从物理层面杜绝任何因应用层漏洞或误操作导致家庭原始数字资产被修改或删除的可能。
• 硬件透传解耦：在 IaC 配置文件（docker-compose.yml）中，严禁硬编码 /dev/dri/renderD128 或特定的 GPU UUID。必须利用 Docker 的 deploy.resources.reservations.devices 特性与探针上报的能力标签进行动态匹配挂载。

### 4.2 硬件拓扑自适应与 I/O 联动大闸
流媒体业务高度依赖底层 I/O 与异构算力。系统通过“探针感知-软开关联动”机制，化解物理硬件插拔带来的系统级异常。

#### 4.2.1 存储介质自适应与降级
• 接入恢复：大容量物理阵列上电后，探针上报 ONLINE 状态。API 网关自动解锁相关功能开关，唤醒关联容器，前端通过 SSE 同步渲染对应媒体入口。
• 缺失熔断：存储阵列意外掉线时，探针触发防抖确认机制。网关锁定控制开关为离线状态，并向底层下发降级指令（pause 或 stop，遵循熔断顺序约束）。（异步极刑规范）：该指令必须包裹 3 秒硬超时；若因 I/O 阻塞导致卡死，立即升级为 SIGKILL 强杀，拦截向系统盘的异常写入。
• UI 隔离：播放中的媒体流平滑中断。前端界面拦截 HTTP 报错，替换为带有磨砂滤镜的缓存骨架屏，并渲染“设备维护中”提示信息。

#### 4.2.2 异构算力节点自适应
• 降级预案：异构加速卡因过热或驱动异常离线时，后端触发“编解码降级预案”，自动将转码策略回退为 CPU 软件解码，保障流媒体播放不中断。
• 探针同步将控制台算力开关重置为 OFF 并推送硬件告警。硬件恢复后，点击开关即可重新加载驱动并恢复硬件加速。

【技术实施细节与路线说明】
• 强制熔断顺序约束：在硬件缺失响应中，系统必须严格遵循第 1 部分定义的熔断顺序：① API 熔断（网关立即返回 503，拒绝新请求）；② 网络断流（从内部服务发现中摘除故障路由，或通过反向代理屏蔽转发）；③ 容器降级（执行 docker pause 或 docker stop -t 10，具体区分见第 1 部分规范）。严禁直接对容器下发暂停指令而不先进行流量阻断。
• CPU 降级调度逻辑与过载保护：当探针确认 GPU 离线后，API 网关必须立即接管。在修改转码配置前，网关必须检查当前 CPU 的实时负载（如利用率 > 80%）。若 CPU 算力充足，网关通过流媒体服务的 REST API 将转码配置（Transcoding Profile）从 nvenc/qsv 修改为 none（或 x264 CPU 软解），实现重试客户端的无感算力漂移。若 CPU 过载，则直接拒绝播放请求，保护主节点不被雪崩拖垮。
• 挂起状态清洗：SIGKILL 强杀容器后，系统会遗留无状态的僵尸容器。网关在执行强制清理时，必须连带调用 docker rm -f，以便在硬件接入时通过 IaC 编排引擎拉起全新且干净的容器实例。

### 4.3 跨平台生态兼容与离线保障
针对移动端操作系统的碎片化，架构在前端代理层提供深度适配策略。

#### 4.3.1 定制安卓系统适配
• 采用 HTTPS 隧道直连并以 PWA 形式添加至桌面。应用运行于独立的 Webview 沙盒进程中，规避定制系统内置浏览器的视频组件劫持（如强插广告、悬浮窗），同时免疫激进的后台进程清理机制。

#### 4.3.2 苹果 iOS 生态适配
• 深度集成 Safari 原生 PWA 特性。媒体流由后端统一转码为 HLS (.m3u8) 格式，前端直接调用 iOS 设备的底层硬件解码器，大幅降低播放功耗。调用 WebAuthn 标准接口实现 Face ID/Touch ID 无密鉴权。

#### 4.3.3 原生安卓系统适配
• 依托完整 GMS 服务，深度接入 Media Session API。在锁屏界面与车载蓝牙设备上自动渲染播放海报与标题，支持物理按键的播放控制。支持通过 Web Push API 接收后台安防告警推送。

#### 4.3.4 局域网离线保障
• 外部公网中断时，前端 PWA 可通过预置的固化静态 IP 或 mDNS 协议直接连接内网主节点。在完全断网状态下，利用 IndexedDB 渲染界面骨架，避免应用白屏。

【技术实施细节与路线说明】
• Service Worker 视频流拦截陷阱 (Range Requests)：对于视频大文件（尤其是 .ts 或 .mp4），前端的 Service Worker 默认无法正确处理带有 Range: bytes=... 的 HTTP 请求，这会导致视频在 iOS 等设备上无法拖拽进度条甚至拒绝播放。必须在 Workbox 中显式引入 workbox-range-requests 插件，并对多媒体路由声明专用的转发规则，确保 206 Partial Content 响应能正确透传至 video.js 播放器。
• PWA 静默热更新防死锁：针对定制系统持久化沙盒的特性，Service Worker 的注册配置必须强制实装 skipWaiting 与 clientsClaim。确保新的前端部署产物下发后，系统能立刻接管控制权刷新缓存，防止长辈的设备永久陷入旧版 UI 的“缓存毒药”中。
• iOS 音频后台驻留：为确保系统在 iOS PWA 下切换至后台时，音频播放与媒体控制不被系统挂起，必须在 DOM 的 <video> 或 <audio> 标签中显式添加 playsinline 属性，并正确注册 navigator.mediaSession 钩子。

### 4.4 基于角色的视图管控与隐私隔离 (RBAC & Privacy)
系统对功能入口进行基于角色的权限过滤与视图重组。

#### 4.4.1 视图分级路由
• 简化模式：过滤系统级监控指标，以大字体、高对比度布局渲染核心媒体与语音交互模块，满足无障碍适老化标准。
• 未成年人模式：结合账号策略触发内容过滤大闸，拦截非适宜内容的 API 路由；将 AI 助手提示词重置为教育模式，并限制知识库检索边界。

#### 4.4.2 情感资产沉淀与物理隐私阻断
• 视觉张量模型侦测到特定事件（如微笑、聚餐）时，支持自动截取视频流归档至相册（此功能默认关闭，受严格授权管控）。
• 隐私物理挂起：提供系统级隐私阻断软开关。触发后，后端瞬间挂起相关的视觉模型推理进程及流媒体录制，释放算力并确保特定场景下的绝对隐私。

#### 4.4.3 统一物联管控
• 通过 MQTT 接入的环境设备统一整合至控制台面板。支持根据语音识别模型的自然语言输入，由后端网关解析并向总线下发设备控制指令。

【技术实施细节与路线说明】
• RBAC 中间件验证：角色的分级路由不仅存在于前端隐藏。后端 FastAPI 必须实装基于 JWT Claims（如 scope: child 或 scope: elder）的 RBAC 拦截中间件。前端发起的任何非授权越权 API 调用，网关直接返回 403 Forbidden。
• 情感资产隔离挂载：自动截取的高频短视频素材不得混写入核心系统卷。必须在 docker-compose.yml 中为其分配独立的、受限额管理的持久化卷（如设定清理阈值或接入生命周期管理 Worker），防止误识别暴增撑爆系统根目录。
• 隐私阻断的物理级切断：当“隐私阻断软开关”开启时，系统不得仅在前端隐藏画面。API 网关必须执行两步底层操作：1. 调用反向代理引擎（如 Caddy/Nginx）的 API，直接熔断并抛弃映射该摄像头的 RTSP/WebRTC 视频流转发；2. 向底层的 AI 监控引擎（Frigate）下发 MQTT 指令或 API，彻底关闭该通道的 Detect 与 Record 进程。确保物理级的绝对截断。

## 第 5 部分：全域可观测性、持续交付流水线与软硬联动矩阵 (V1.58 终极满血版)

### 5.1 运维与可观测性技术路线规约
为确保系统在长生命周期内的稳定性与排障能力，可观测性层与部署流水线强制采用云原生标准组件，并实现监控体系与业务容器的彻底解耦。

#### 5.1.1 指标采集与时序数据库 (Metrics & TSDB)
• 采集引擎：采用 Prometheus 作为核心指标采集与时序数据存储引擎。
• 采集范围：包含宿主机 CPU/内存/网络指标、异构计算卡（GPU/TPU）利用率及温度、存储阵列 I/O 吞吐、以及所有业务微服务容器的运行状态。
• 服务发现：通过文件服务发现（file_sd_configs）或 DNS 记录动态添加监控目标，新增节点自动纳入监控，无需手动重启服务。
• 存储寿命保护：Prometheus 的 TSDB 数据目录必须强制挂载至独立 HDD（机械硬盘）或 tmpfs（内存文件系统），严禁使用系统 SSD 进行高频监控数据写入，以防固态硬盘寿命过快损耗（遵循第 1 部分 F-08 实装）。

#### 5.1.2 可观测性采集器选型 (Telemetry Agent)
• 统一采集器：强制采用 Categraf 替代碎片化的 node_exporter、cadvisor 等组件，降低节点资源开销。
• 高基数防御：在 categraf.toml 中显式配置标签白名单，仅保留 job、instance、node 等静态标签，默认丢弃 container_id、trace_id 等高基数动态标签，防止时序数据库内存爆炸。
• 职责分离：Categraf 仅负责周期性指标的被动采集；而 Topology Sentinel（拓扑探针）负责实时的硬件热插拔检测与应急熔断。两者部署互不依赖，符合第 1 部分探针职责边界。

#### 5.1.3 Docker Socket 只读代理控制 (安全大闸)
• 安全漏洞封堵：严禁将宿主机的 /var/run/docker.sock 直接挂载至 Categraf 或其他监控容器中，以防监控组件被攻破后引发宿主机提权。
• 代理层引入：额外部署专用的 docker-socket-proxy 容器代理底层 Socket。配置严格的 HTTP API 路由控制，仅放行 GET /containers/json 等只读请求，从网关层拒绝一切 POST、DELETE 写操作。监控组件通过 TCP 协议连接该代理容器获取运行状态。

#### 5.1.4 数据可视化与日志聚合
• 看板展示：采用 Grafana 渲染监控仪表盘。所有仪表盘配置通过 JSON 模板生成并纳入 Git 版本控制。
• 日志聚合：采用 Promtail + Loki 轻量级日志栈。Loki 存储目录同样需挂载至 HDD。日志默认保留 7 天。

#### 5.1.5 CI/CD 交付与编排 (GitOps)
• 配置即代码：以 Git 仓库为唯一事实数据源，包含 system.yaml、环境密文及监控规则。
• 自动化部署与回滚：通过幂等的跨平台部署脚本（deployer.py）执行隧道通信与容器重载。部署脚本必须支持回滚指令（如 --rollback），通过切换历史配置快照实现故障秒级恢复。

【技术实施细节与路线说明】
• Socket 代理挂载规约：监控容器（Categraf）的 docker-compose.yml 编排中，必须注入环境变量 DOCKER_HOST=tcp://docker-proxy:2375，严禁出现 volumes: ["/var/run/docker.sock:..."] 的挂载语句。
• 点火与交付边界划分：运维团队必须严格区分：bootstrap.py 仅限“第 0 天”裸机环境预检与初始拉起使用；系统上线后的任何服务更新、监控规则修改与回滚，必须且只能通过触发 deployer.py 结合配置编译器完成。
• 时序库与日志源防爆大闸：若 Prometheus 使用 tmpfs 挂载，启动参数必须强绑定上限阈值（如 --storage.tsdb.retention.size=1GB），严防内存泄漏引发宿主机 OOM。同时，全局 daemon.json 必须配置 Docker 日志轮转（如 max-size: "50m", max-file: "3"），在被 Promtail 抽取前防止“日志风暴”直接撑爆磁盘。

### 5.2 硬件拓扑自适应与联动控制方案
针对可关闭软件功能的动态开关，实施从前端 UI 到宿主机内核的物理级防死锁联动。

#### 5.2.1 功能开关的底层控制逻辑
• 用户在前端发起开关切换请求，经加密隧道送达 API 网关并进行最高指挥官权限校验。
• 【强制熔断顺序贯彻】：API 网关不得直接调用容器 SDK。必须严格按序执行：① 自身 API 层熔断拦截（返回 503 Service Unavailable 或 202 Accepted）；② 从服务发现/代理层物理摘除路由（如从 Consul 注销，或更新 Nginx upstream）；③ 向底层下发容器降级指令（依据容器类型执行 docker pause 或 docker stop -t 10）。
• 极刑防死锁：执行暂停操作（docker pause）强制包裹 3 秒异步超时，超时则升级为 SIGKILL 强杀容器。状态写入 Redis，并通过 SSE 广播至全域客户端，同步更新面板开关的 UI 状态。

#### 5.2.2 应对异常缺失的降级大闸
• 物理离线保护：探针确认外部存储等核心硬件意外断开后，直接触发网关防爆机制。将控制开关重置为锁定状态，并严格触发『① API 层阻断拦截 → ② 网络层路由摘除 → ③ 容器级指令下发』的三步标准熔断流程，彻底拦截后续向丢失挂载点发起的读写请求。
• 热阻断降频大闸：探针持续监控算力节点温度。当侦测到核心温度超限（如 >85°C，阈值可配置）时，系统自动触发降频策略。网关必须严格执行三步熔断顺序：首先在 API 层抛出 503 拒绝为该节点分配新任务；其次在内部网络路由中物理摘除该节点的请求转发；最后向底层下发指令，静默挂起（docker pause）非核心的高负载推理或转码容器。待温度回落后自动解冻恢复。

#### 5.2.3 应对硬件接入的恢复流程
• 探针检测到新设备或算力节点通电接入，完成三次防抖校验后更新状态树。控制面板对应的开关解除锁定。
• 用户点击启动后，系统下发 docker restart 命令重新构建容器的文件描述符映射，网络路由同步恢复，业务无损拉起。

【技术实施细节与路线说明】
• 可观测联动闭环：当控制开关触发熔断（无论是手动还是探针被动触发）时，Prometheus 必须通过拉取 Categraf 指标（如 docker_container_status 变为 paused 或 exited）自动捕捉状态变化。Alertmanager 需配置相关规则，将“计划内维护（手动触发）”与“意外宕机（硬件掉线）”进行告警抑制区分。
• 熔断顺序一致性声明：本部分所有“熔断”动作必须严格遵守第 1 部分第 1.3 节定义的顺序，严禁跳过网络断流直接操作容器。开发实现时需通过代码审查确保此顺序不被破坏。

### 5.3 跨平台告警分发与触达网络
告警系统需根据不同移动操作系统的特性，执行定制化的分发策略，确保消息高可用触达。

#### 5.3.1 定制安卓系统告警策略
• 系统级触达：针对激进的后台清理机制，利用第三方 Webhook（如 Server酱、PushPlus）对接微信公众号的模板消息接口，绕过系统进程限制实现告警推送。
• 严重告警升级：对于最高级别（CRITICAL）的物理级灾难告警（如主盘离线、节点失火），触发语音通知网关 API 进行电话轮播呼叫。

#### 5.3.2 苹果 iOS 生态告警策略
• 私有推送链路：部署自建的 Bark 容器化推送服务端，确保敏感告警数据完全私有。支持通过 iOS 系统级通道下发带有自定义铃声和震动强度的通知。
• 状态推送：非紧急状态更新（如增量备份完成、模型拉取成功）通过标准 Web Push API 静默推送至通知中心。

#### 5.3.3 原生安卓与桌面端告警策略
• 双向交互通信：集成 Telegram Bot 协议。支持在聊天软件内接收告警，并允许具有最高权限的用户通过回传指令（点击消息按钮）远程下发重启容器或挂起任务的控制命令。

【技术实施细节与路线说明】
• 【高危交互的 2FA 拦截大闸】：针对 Telegram Bot 的双向操作（如点击“重启服务”或“停止转码”），系统严禁直接执行。网关接收到 Webhook 回调后，必须向用户下发一次性动态验证码（OTP）或要求输入预设的静态高危密码进行二次质询（2FA）。该 2FA 会话状态必须设定极短的 TTL（如 60 秒），逾期未收到正确回复则自动作废该上下文，彻底防御设备丢失或 Telegram 账号被劫持引发的系统沦陷及内存状态堆积。

### 5.4 架构演进能力
• 微服务组件的无损扩容：得益于 Prometheus 的服务发现机制，新增的业务容器将自动被系统抓取监控指标。通过关联 Grafana 的正则匹配模板，无需手动编写任何采集规则即可在仪表盘中自动长出新组件的监控图表。
• 动态配置同步：新增服务在 IaC 配置表（system.yaml）中声明后，系统能力矩阵随之更新，前端自动按协议渲染新模块的控制卡片及状态灯，实现全链路的“零代码修改”横向扩展。

## 第 6 部分：跨端交互、离线容灾防御与协议驱动渲染层 (V1.58 终极满血版)

### 6.1 前端核心技术路线规约
为确保前端应用在较长生命周期内具备高可维护性与平滑演进能力，前端渲染层必须彻底与底层硬件环境解耦。系统严禁依赖特定操作系统应用商店的原生客户端（Native App）开发模式。

#### 6.1.1 核心渲染与构建引擎
• 框架选型：必须采用 Vue 3 (Composition API) 结合 Vite 构建工具。
• 工程化规范：组件必须按业务域进行模块化拆分，并强制启用按需加载（Code Splitting）策略。推荐依赖 Vite 的极速冷启动与虚拟 DOM 优化，以保障低算力移动设备上的首屏加载性能（若采用其他打包策略，需提交 ADR 说明）。

#### 6.1.2 跨端交付机制 (Delivery Protocol)
• 标准规范：必须严格遵循 PWA (Progressive Web App) 渐进式 Web 应用规范。
• 运行形态：必须通过配置 manifest.json 与沉浸式全屏声明，允许用户将应用直接添加至设备桌面。应用将以独立进程运行，消除浏览器地址栏等冗余外壳。
• 离线代理：必须注册 Service Worker，拦截全域网络请求并执行资源缓存策略，作为断网环境下的第一道物理防线。

#### 6.1.3 离线存储架构 (Offline Storage)
• 缓存策略引擎：推荐引入 Workbox 自动生成 Service Worker 的静态资源与高频 API 数据缓存策略。
• 本地数据库：建议采用 Dexie.js（对浏览器原生 IndexedDB 的工业级封装）作为本地状态存储。缓存内容必须包含最近的安防快照、健康指标摘要、多媒体海报元数据以及硬件开关的末次状态快照。
• 数据持久化授权：为防止移动端操作系统在存储空间紧张时自动清理数据，前端应用初始化时必须调用 navigator.storage.persist() 接口向用户申请持久化存储配额。若浏览器拒绝持久化授权，建议在前端 UI 抛出“离线降级可用性受损”警告（忽略此警告需登记 ADR），避免静默失败导致断网骨架失效。

#### 6.1.4 [实装 F-04] 零信任鉴权与断网降级机制
• 无密码鉴权标准：必须全面接入 WebAuthn (FIDO2) 协议。利用终端设备的独立安全芯片，实现高强度的本地活体认证验证。
• 离线降级访问令牌：用户首次成功联网登录时，系统必须下发一份长有效期、极低权限的离线专用 JWT。该令牌必须通过 Web Crypto API 加密存入本地 IndexedDB。断网唤醒时，直接解密该令牌进入“只读骨架屏模式”，严禁在离线态开放任何数据修改权限。
• 局域网 PIN 码降级方案：针对生物特征模块损坏场景，提供基于局域网 IP 校验的 6 位数字 PIN 码备用通道。严禁使用软弱的速率限制，必须严格对齐底层红线：“连续失败 5 次，强制锁定请求 IP 15 分钟”。通过此通道登录的会话，必须被强行限制访问高危控制面板。

#### 6.1.5 Schema 驱动渲染引擎 (Schema-Driven UI)
• 动态视图生成：严禁在前端硬编码任何功能按钮及子路由。所有 UI 组件库必须由后端下发的 JSON Schema 协议动态驱动。
• 能力矩阵消费：前端应用必须通过定时轮询或 SSE 获取 /api/v1/capabilities 返回的能力矩阵，利用 v-for 结构自动渲染对应的服务卡片与状态开关。

### 6.2 状态同步与 UI 容灾闭环
UI 界面作为底层硬件拓扑状态的实时物理映射，必须具备处理 I/O 阻塞与状态缺失的能力。

#### 6.2.1 应对硬件缺失的 UI 优雅降级
• 状态阻断响应：底层挂载点或外设断联后，探针触发后端冻结逻辑，并通过 SSE 推送 STORAGE_OFFLINE 状态变更至前端。
• 前端物理锁定：前端接收到离线事件后，对应的功能开关必须自动切换至 OFF 状态，并附加不可交互的置灰锁定样式。此举从 UI 层彻底切断无效的 I/O 重试请求。
• 无感异常隔离与极客排障透传：针对多媒体播放等业务模块，前端必须拦截后端 502/503 报错，从 IndexedDB 提取缓存数据渲染骨架屏。必须根据账号角色（RBAC）差异化渲染：普通用户显示“设备维护中”的磨砂提示；指挥官/管理员视图下，必须暴露真实的 503 状态码与 TraceID 供排障使用。

#### 6.2.2 应对硬件接入的 UI 状态恢复
• 自动解锁：新硬件通电或重连后，探针上报 ONLINE 状态。前端面板的控制开关必须同步解除锁定限制，恢复至可交互的待命状态。
• 平滑重载：用户触发开启指令后，底层解冻容器，前端对应模块的限制滤镜自动移除，业务入口推荐实现无刷新加载。

#### 6.2.3 [实装 F-01] 前端侧断路器 (Client-Side Circuit Breaker)
• 当后端 Redis 宕机或发生真实 I/O 阻断引发后端的“悲观静默”时，前端严禁执行“乐观放行”的盲目重试逻辑。
• 前端一旦收到后端的 503 (Service Unavailable) 熔断信号，必须立即触发本地断路器，强制停止对该故障模块的所有异步轮询或心跳探测。彻底配合后端的“悲观静默”策略，保护后端协程池不被重试风暴耗尽。

#### 6.2.4 渲染防抖控制
• 全链路防抖：前端在接收 SSE 推送时，必须实装节流与防抖（Debounce）函数。严禁因网络波动或接口物理接触不良导致短时间内进行高频 DOM 重绘与页面闪烁。

### 6.3 跨平台环境深度适配
针对不同移动操作系统的技术限制与特性，实施差异化的生态兼容方案。

#### 6.3.1 定制安卓系统适配策略
• 规避内置劫持：推荐通过 HTTPS 隧道结合 PWA 沙盒运行模式，阻断定制系统内置浏览器对 <video> 标签的广告注入与悬浮窗劫持。
• 进程存活保障：独立 PWA 进程不依赖后台常驻 VPN 软件，可免疫各大定制系统激进的后台资源清理策略。

#### 6.3.2 苹果 iOS 生态适配策略
• 原生组件调用：网关下发媒体流时，必须适配 iOS Safari 的原生 HLS (.m3u8) 流。前端必须穿透调用 iOS 系统的硬件媒体引擎进行解码，保障高码率视频的低功耗播放。
• 系统级权限集成：必须深度集成 WebAuthn，并推荐接入 iOS 锁屏播放控制功能，提供与原生 App 一致的操作体验。

#### 6.3.3 原生安卓系统集成策略
• 系统级 API 调用：在具备完整 Chrome 内核的设备上，建议接入 Media Session API。可在车载蓝牙终端与系统下拉通知栏渲染媒体控制总线。
• 后台异步通信：可选结合 Web Push API 与 Background Sync API，实现离线状态下的安防告警推送与网络恢复后的数据静默回传。

#### 6.3.4 极限断网应急保障
• 局域网物理路由：应对外网光猫故障等断网场景，推荐基于 mDNS 协议直接连接内网主节点。可选自动生成包含本地静态 IP 的直连二维码，辅助移动端通过局域网路由进行紧急抢修。

### 6.4 架构演进与安全合规管控
UI 架构必须严格遵守零代码修改扩展与数据隐私合规标准。

#### 6.4.1 功能模块动态扩容
• 无感横向扩展：引入新硬件设备或微服务时，系统能力矩阵更新后，各终端前端面板必须能够自动解析并动态挂载对应的控制组件与路由，实现零前端代码重构的迭代演进。

#### 6.4.2 敏感操作管控与数据合规
• 二次身份核验：针对高危操作（如格式化阵列、擦除历史数据等），前端在下发 API 请求前必须强制二次唤醒 WebAuthn 进行活体确认（或要求输入高危 PIN 码）。严禁拦截失败的非授权请求向底层透传，确保与网关的 RBAC 中间件形成多层防护。
• 数据主权与终局处理：必须内置符合标准的“数据可携带权”功能模块，支持按账户提取全量日志与资产（JSON/CSV 格式）。
• 视觉隐私屏蔽：可选集成轻量级 Canvas 绘图组件，支持用户在安防快照画面中框选隐私排除区域。前端将其转化为多边形坐标数组上报后端，指示视觉推理模型执行屏蔽过滤。

## 第 7 部分：全局初始化部署与容灾恢复体系 (V1.58 终极满血版)

### 7.1 全局初始化技术路线规范 (Bootstrapping & IaC)
为确保系统在多节点环境下的标准化交付与后续维护的强一致性，基础设施的部署必须严格遵循 IaC（基础设施即代码）理念。严禁开发与运维人员在宿主机环境中执行非标准化的手动配置命令。

#### 7.1.1 核心编排引擎
• 引擎选型：必须依赖 Docker Daemon 与 Docker Compose 声明式语法（v3.8+）。
• 部署规范：所有系统服务及后台任务必须实现 100% 容器化，依靠容器引擎提供底层硬件与上层微服务之间的隔离层与资源配额控制。

#### 7.1.2 环境变量与敏感凭证管理
• 环境注入：必须使用 .env 文件配合高强度密文管理机制（推荐使用 Docker Secrets，或 HashiCorp Vault 的轻量级替代方案，若降级使用明文 .env 需提交 ADR 豁免）。
• 代码防腐：所有数据库密码、API 密钥、外部网络隧道的鉴权 Token 严禁硬编码于代码库中。初始化向导必须支持自动生成高强度的随机密码与 JWT 密钥。

#### 7.1.3 自动化初始化脚本 (Bootstrap Process)
• 核心职责：跨平台的初始化脚本（如 bootstrap.py）作为系统“第 0 天”启动的唯一入口，必须按序执行以下标准化流程：
1. 环境基线预检（Docker 版本、端口冲突、磁盘配额、内核参数）。
2. 物理数据卷的挂载点核验与权限修复。
3. 加密网络隧道（Headscale/Cloudflare Tunnel）的优先拉起。
4. 执行微服务阵列的点火命令 (docker-compose up -d)。
• 执行幂等性：该脚本必须保证幂等操作，重复执行时仅应用增量变更，严禁因重复运行破坏系统当前存量状态。

#### 7.1.4 供应链多源高可用防线
• 多源配置拉取：初始化脚本必须内置容灾网络逻辑。若主代码托管平台（如 GitHub）发生访问阻断，必须能够自动切换至备用源（如 Gitee、自建 Gitea 节点）拉取最新配置文件。
• 私有镜像仓库：核心服务镜像建议托管于自建的私有仓库（如 Harbor），并推荐配置定时同步任务，确保断网或外部镜像站封锁条件下的系统重建能力。
• 完整性校验：所有拉取的配置文件与容器镜像，必须通过 Checksum 或签名验证，防止遭遇中间人篡改。

#### 7.1.5 高级配置选项 (Optional Features)
• Shamir 秘密共享：可选支持将最高权限密钥（如 Headscale 根凭证、Restic 备份密码）通过 Shamir 算法分割为多份保管，规避单点密钥丢失风险。
• 系统配置自动迁移：部署脚本推荐内置配置版本检测机制，旧版 system.yaml 可自动调用迁移脚本无损转换为新版格式，并生成审计日志。

### 7.2 硬件拓扑自适应与异常联动控制
在分布式与异构硬件环境中，系统必须具备对网络分区与物理硬件异常的自动响应能力，防止数据污染与级联崩溃。

#### 7.2.1 边缘节点网络分区静默待命机制 (Split-Brain Prevention)
在多节点集群模式下，针对边缘工蜂节点与主节点断联的场景，必须实装防脑裂大闸：
• 心跳监测：边缘节点内部运行轻量级 Watchdog 守护进程，持续向主控节点 Redis 写入心跳。
• 网络分区熔断：若连续 30 秒无法完成心跳同步（判定为失联），边缘节点严禁执行任何主动杀容器的动作（如 docker pause）。节点必须立即降级为“静默待命”状态，停止从队列拉取新任务 (Stop-Pulling)。
• 状态恢复：网络恢复且与主节点完成状态重同步后，节点必须接受主控节点的仲裁，上报滞留任务的缓存结果并恢复队列轮询。

#### 7.2.2 异常状态熔断与温控保护
• 热阻断机制：拓扑探针持续探测异构计算卡及主板的底层温度（S.M.A.R.T. 与 nvml 接口）。当温度突破安全阈值（如 >85°C）时，探针必须向 Redis 写入高温警报状态，严禁探针越权直接操作底层容器。
• UI 状态与物理熔断同步：API 网关接管警报后，必须严格按序执行三步熔断：① 拦截相关服务调用（API 熔断返回 503），前端对应软开关强制锁定为 OFF；② 从内部网络代理中物理摘除该异常节点；③ 向底层下发 docker pause 指令，安全挂起高负载的 AI 推理与视频转码容器，防止硬件烧毁。

#### 7.2.3 硬件恢复与远程唤醒 (WoL)
• 网络唤醒：宿主机因市电中断彻底关机后，可选支持通过局域网内其他常备低功耗设备发送 Magic Packet 唤醒核心节点。
• 平滑接管：硬件重新上线并降温后，探针上报恢复状态。前端软开关解除锁定，用户点击开启后，系统必须下发 docker restart（严禁使用 unpause）重新加载硬件驱动及文件描述符，业务无损拉起。
• 状态防抖：底层必须实装滑动窗口（连续 3 次心跳一致）防抖算法，避免物理接口接触不良导致的系统状态频繁翻转。

#### 7.2.4 人工维护模式
建议支持运维人员通过系统控制台主动将特定硬件模块的开关置为 OFF，系统将执行缓存落盘与容器安全挂起，保障物理热插拔维护过程中的数据安全。

### 7.3 跨平台设备交付与兼容性保障
系统基于 PWA 架构，针对不同操作系统的底层限制，执行标准化的交付与兼容策略，确保各端操作体验一致。

#### 7.3.1 定制化安卓系统适配
• 推荐摒弃常驻后台的代理应用，采用云端加密隧道结合 PWA 桌面独立进程运行。彻底规避定制操作系统（如 HarmonyOS、MIUI 等）对后台进程的激进清理策略，以及内置浏览器对媒体播放组件的劫持行为。

#### 7.3.2 苹果 iOS 设备生态适配
• 必须深度集成 Safari PWA 特性，调用底层 WebAuthn 实现硬件级安全解锁。媒体流分发必须采用原生支持的 HLS 协议，穿透调用苹果芯片（A/M 系列）的硬件解码模块，实现高码率视频的低发热与低功耗播放。

#### 7.3.3 原生安卓与桌面极客端适配
• 推荐深度接入系统原生 API。配合主权加密虚拟局域网（Headscale），桌面端可直接利用物理内网带宽实现异地无延迟运维与原盘文件挂载。移动端建议集成 Web Push 与 Media Session 接口，实现系统级消息推送与锁屏媒体控制。

### 7.4 架构演进能力评估
全栈解耦的设计确保了系统在未来长周期内的技术栈更迭与硬件升级不受逻辑阻碍。

#### 7.4.1 容器化底座的平滑升级
由于应用层与系统环境通过容器镜像 100% 解耦，前端采用静态 Schema 驱动渲染，后端采用微服务 API 网关。若需升级核心框架版本或替换后端语言，仅需变更相关服务的 Docker 镜像标签，实现基础设施零代码重构的平滑演进。

#### 7.4.2 功能模块的动态扩展
系统严禁与特定的娱乐或功能模块硬绑定。新增边缘算力节点或第三方开源服务（如云游戏串流容器）时，必须做到仅在环境配置中声明路由规则。前端跨端设备将根据下发的 JSON 协议自动渲染对应的交互组件，调度网关自动纳管算力。

#### 7.4.3 硬件抽象与自适应适配
探针机制必须仅识别硬件的“能力标签”（如编码格式支持、计算张量类型），严禁硬编码具体硬件型号。硬件升级（如显卡架构更迭）完成后，探针重新上报硬件能力矩阵，任务调度器自动进行权重适配，核心代码层无需干预。

## 第 8 部分：基础设施即代码 (IaC) 与持续交付管线 (V1.58 终极满血版)

### 8.1 基础设施即代码与配置管理规范
为确保系统在长生命周期内的可维护性与多节点环境下的强一致性，基础设施部署必须强制采用 IaC（Infrastructure as Code）理念。严禁开发与运维人员在宿主机或代码库中进行任何非追踪的硬编码或手动变更操作。

#### 8.1.1 单一事实来源 (Single Source of Truth)
• 配置收束：必须采用结构化标记语言（YAML）。后端的微服务编排、硬件算力分配、网络隧道映射，以及前端应用视图的组件开关布局，必须统一收束至全局配置文件（system.yaml）中。
• 版本溯源：系统的全量运行状态必须由 Git 仓库中的特定 Commit 唯一确定，彻底解决“配置漂移”与“环境不一致”问题。

#### 8.1.2 模块化配置编译器 (The Compiler)
• 编译引擎：必须采用 Python 与 Jinja2 模板引擎构建配置编译器。严禁在 Jinja2 模板中混入复杂的 if-else 业务判断，其职责仅限于纯粹的变量替换与渲染。
• 模块化聚合：为避免单体配置文件过大，推荐支持读取 conf.d/ 目录下的碎片化 YAML，在内存中合并校验后统一处理（若团队决议仅维护单文件，需提交 ADR 豁免）。
• 自动化拓扑生成：编译器在部署前，必须根据 YAML 配置自动渲染生成所有下游环境文件：
• 后端环境密文（.env）与容器编排文件（docker-compose.yml）。
• 反向代理路由规则（Caddyfile / nginx.conf）。
• 监控与告警采集规则（prometheus.yml 等）。
• 前端控制台所需的 JSON Schema 协议（路由与组件挂载逻辑）。
• 【编译校验大闸】：部署器在执行上述产物前，必须强制挂载 config-lint（YAML 语法与 Schema 校验）。一旦校验失败，部署流水线必须立即阻断并抛出异常。

#### 8.1.3 幂等部署与边缘分发 (Idempotent Deployment)
• 部署器机制：部署脚本（deployer.py）必须具备绝对的幂等性。重复执行时仅应用增量变更，严禁因重复触发而破坏当前存量运行状态。
• 多节点下发：在联邦算力集群中，可选通过 SSH 协议自动向未知的边缘工蜂节点下发探针程序与环境配置。若启用此功能，边缘节点启动后必须强制向主节点注册并服从全局调度，严禁保留独立自裁决权。

#### 8.1.4 [实装 F-09] 架构决策记录 (ADR) 管理规范
• 决策追溯：为解决系统长期演进中“历史技术决策无法追溯”的问题，必须引入 ADR (Architectural Decision Records) 机制。
• 执行标准：任何涉及核心技术栈更迭、API 契约修改、硬件抽象层重构或偏离“建议级”规范的重大变更，必须在合并代码前，于代码库 docs/adr/ 目录下提交 Markdown 格式的 ADR 文档。文档必须包含：背景上下文、决策选项、评估对比、最终决定及影响范围。

#### 8.1.5 离线种子包与冷启动恢复 (Offline Seed & Cold Start)
应对极端物理断网（“第 0 天”无外部互联网接入）场景，构建全量离线恢复机制：
• 数据快照导出：必须通过 export-seed.sh 脚本定期提取 Git 仓库快照、依赖容器镜像（通过 docker save 导出为 .tar），以及离线部署脚本。
• 物理加密与加载：将离线包储存于独立物理介质（如 U 盘）时，导出脚本必须调用底层加密工具（如 LUKS 或 VeraCrypt）对数据实施块级加密，防止包含凭证的物理遗失。在无公网环境下执行 bootstrap-offline.sh，脚本自动解密并执行镜像导入、配置编译与容器拉起，实现一键闭环重建。

### 8.2 硬件拓扑自适应与 UI 渲染的 IaC 闭环
系统的 UI 组件由配置编译器动态生成，实现前端展现与底层硬件的工程解耦。

#### 8.2.1 硬件能力的动态声明
• 标签化路由：在 system.yaml 中严禁出现具体硬件型号的声明，必须采用“能力（Capabilities）”标签（如 vision, gpu_nvenc_v1）。编译器根据节点能力自动映射 API 路由策略，并同步生成前端控制台的功能矩阵配置文件。

#### 8.2.2 状态流转与控制链路
• 故障响应：底层挂载点或算力资源异常时，探针触发网关冻结机制。由编译器生成的前端软开关必须切入“只读离线”锁定状态。后端的控制链路必须严格执行“① API 层熔断 → ② 网络层路由摘除 → ③ 容器级指令下发”的三步降级规范。
• 恢复响应：新硬件接入并完成三项物理校验后，探针上报存活，UI 开关恢复可交互状态。前端触发开启指令后，容器重新拉起并构建挂载映射，系统无损恢复。

#### 8.2.3 状态防抖与操作拦截
• 硬件防抖：探针层必须执行滑动窗口算法（连续 3 次探测一致），过滤物理接口微小接触不良导致的误判。
• 操作鉴权：由 IaC 生成的高危功能按钮（如存储卸载、重置），在前端触发前必须绑定强制性的 WebAuthn 二次活体验证（或高危 PIN 码质询）逻辑。严禁任何未经 2FA 的高危请求穿透至网关层。

### 8.3 跨生态应用打包与交付自动化
系统无需为各操作系统单独编译应用分支，由配置编译器自动生成跨端自适应协议。

#### 8.3.1 定制安卓生态编译策略
• 编译器在构建阶段必须自动化生成 PWA manifest.json 与基于 Workbox 的 Service Worker 缓存配置。应用作为独立沙盒运行，切断内置浏览器对多媒体播放器元素的 DOM 级劫持与重写，实现标准化部署。

#### 8.3.2 苹果 iOS 生态编译策略
• 编译器输出配置时，必须强制下发适配 Safari 引擎的原生 HLS 媒体流与 WebAuthn API 调用。前端必须穿透调用 iOS 设备的底层 A/M 系列硬件解码器完成计算任务，严禁回退至高功耗的软件轮询渲染。

#### 8.3.3 断网骨架屏预编译
• Dexie.js 离线数据库模型及基础骨架屏逻辑必须在编译期即被注入前端产物中。遭遇主节点失联时，前端自举接管路由并展示离线提示。
• 可选在离线屏中动态生成包含设备最后已知局域网静态 IP 的二维码，以辅助用户或运维人员进行网络抢修（防止过度设计，视开发资源排期）。

### 8.4 系统版本化演进控制
基础设施版本化保障了系统横向与纵向演进过程中的绝对安全。

#### 8.4.1 配置驱动的动态扩容
• 零代码重构：如需新增本地大模型引擎或新的多媒体微服务，必须仅需在全局配置表（system.yaml）的 services 节点下增加声明。编译器执行后，自动变更前端路由映射并挂载新的控制组件，监控端点同步纳管，确保前后端核心业务的代码修改率为 0。
• 硬件平滑更迭：探针运行时必须仅识别并上报硬件编码格式等属性标识。硬件升级（如更换新架构显卡）引发的能力变更，必须由系统自动匹配下发调度任务，核心代码层对具体物理变更保持零感知。

#### 8.4.2 GitOps 与秒级状态回滚
• 声明式溯源：所有架构拓扑与配置文件的迭代必须以 Git 提交树为唯一事实基准。
• 无损回滚：如遇错误的配置下发引发系统级服务不可用，运维人员必须能够通过执行 git revert 回退至前一个健康 Commit 并重新触发流水线。编译器与部署器必须强制将系统重置回上一个绝对健康的物理快照状态，阻断级联失效并避免冗长的故障排查。

## 第 9 部分：分布式联邦集群与算力动态调度架构 (V1.58 终极满血版)

### 9.1 设计背景与架构目标
随着私有云业务的演进，单机节点算力（如单张显卡）将无法满足高并发多模态 AI 推理与多路流媒体转码的复合需求。本架构引入联邦集群模型，将算力向多节点横向扩展。
• 算力抽象化：必须将不同指令集架构（x86/ARM）及异构硬件（带有 GPU/TPU 的边缘低功耗设备）统一抽象为“算力资源池”，基于能力标签进行任务调度。
• 动态弹性伸缩：必须支持边缘节点即插即用，实装节点自动注册与存活探活机制，严禁在日常扩容中人工干预全局配置文件。
• 负载感知调度：必须依据节点的实时 CPU、GPU 及内存负载动态分发计算任务，规避单点性能瓶颈。
• 高可用与自愈：当计算节点发生物理失联时，调度器必须自动触发任务迁移，保障核心业务连续性。

### 9.2 集群拓扑与物理角色定义
集群架构必须采用控制面与数据面分离的设计原则。
• 主控节点 (Control Plane Node)：必须部署于性能最强的核心宿主机。负责运行全局状态机（Redis）、核心 API 网关与任务调度器。
• 边缘工作节点 (Worker Node)：可部署于任意局域网或广域网边缘设备（如闲置 PC、Mac Mini、开发板）。必须负责运行节点探针（Agent），接收调度指令并执行容器化计算负载。
• 共享存储节点 (Storage Node)：可选部署。可通过 NFS 或 Ceph 提供集群级别的共享持久卷，保障有状态任务在节点间漂移时的数据一致性（注：由于家庭网络带宽易形成瓶颈，若采用此方案，强烈建议使用 2.5G/10G 独立物理网络）。

### 9.3 节点生命周期与状态机管理
边缘节点必须通过扩展后的 worker-agent 守护进程接入全局状态机。

#### 9.3.1 注册与心跳上报
• 节点注册：工作节点启动后，必须向主控节点的 /api/v1/cluster/register 端点发起注册请求。必须上报自身的能力标签（如 capabilities: ["cpu", "gpu"]）及物理资源配额（如 vram_mb: 6144）。
• 高频心跳：工作节点必须每 5 秒向主控节点上报一次实时负载指标（包含 cpu_percent, gpu_percent, vram_used_mb 及本地队列长度）。

#### 9.3.2 离线侦测与容灾
• 主脑仲裁机制：主控节点必须对心跳执行滑动窗口监控。若连续 3 次（约 15 秒）未收到心跳包，触发离线判定大闸，将该节点状态标记为 OFFLINE。
• 主控容灾调度：调度器随即必须终止向该节点下发新任务，并将其负责的异步队列任务重新分配至其他健康节点。
• 【绝对红线：剥夺自决权】：当边缘节点发现自身与主控节点断联时，严禁触发任何主动杀容器或业务重启的自裁决动作！节点必须立即降级为“停止拉取新任务（Stop-Pulling）”状态并静默待命，直至网络恢复并重新接受主控节点的仲裁。

### 9.4 任务调度总线与负载均衡

#### 9.4.1 全局能力矩阵 (Capability Matrix)
• 主控节点必须在 Redis 中维护一份全局动态能力矩阵。记录集群内所有可用服务（如 llm, vision, media）及其对应的健康节点端点列表与实时负载权重。前端通过接口拉取该矩阵，实现 UI 控制卡的动态渲染。

#### 9.4.2 加权感知调度算法 (Load-Aware Scheduling)
任务下发时，调度器必须执行以下过滤与加权逻辑：
• 硬性条件过滤：必须筛除不具备对应 capability 标签或剩余显存/内存（vram/ram）不满足任务最低阈值的节点。
• 动态加权排序：推荐针对满足条件的节点集合，按照当前负载进行加权排序（权重公式可配置，如 CPU利用率 * 30% + GPU利用率 * 70%），优先将请求路由至分值最低的最优空闲节点。

#### 9.4.3 任务执行模式
• 同步调用 (Synchronous)：针对低延迟敏感型任务（如轻量级文本推理），主控网关必须直接将 HTTP 请求反向代理至边缘节点的暴露端口，同步等待响应。
• 异步队列 (Asynchronous)：针对重型耗时任务（如批量视频转码），主控节点必须将任务元数据推入 Redis 消息队列。边缘节点以 Pull 模式轮询队列消费任务，执行完毕后必须通过 Callback 接口回传状态。

### 9.5 集群网络安全边界
• 控制面隔离：严禁集群管理端口暴露于公共网络。所有边缘节点必须通过 Headscale/WireGuard 等加密虚拟网络接入主控节点。主控网关及 Redis 必须仅绑定并监听虚拟 IP (VIP)。
• 零信任准入验证：节点注册接口必须强制校验预共享密钥 (Pre-shared Key)。推荐使用 mTLS（双向 TLS）证书实现高强度双向认证，阻断未授权的恶意节点伪装接入集群（若因性能开销或配置复杂度豁免 mTLS，必须提交 ADR 决策记录）。
• 数据面加密：任务数据流转及共享存储挂载必须在内网或加密隧道内完成，保障通信信道防窃听。

### 9.6 与现有 IaC 架构的集成规范
集群架构的扩展必须严格遵守第 8 部分的 IaC 规范。
• 配置编译器扩充：system.yaml 全局配置表必须新增 cluster 结构块。允许声明主控节点端点及静态工作节点的初始化资源配额。
• 部署器自动化：自动化脚本可选通过 SSH 自动拉起边缘节点隧道，并分发编译后的 worker-agent 配置文件，实现新节点的秒级点火与集群扩容。若不使用自动化下发，必须提供一份标准化的手动注册脚本供边缘设备执行。

### 9.7 集群架构已知工程约束 (Backlog)
针对当前 V1.58 版本的集群设计，已在架构决策库中登记以下待优化项，推荐在未来版本迭代解决：
• F-10：多节点依赖单一 NFS 共享存储时，在极高并发读写场景下可能存在 I/O 吞吐瓶颈。
• F-11：当前的调度算法仅依赖瞬时负载采样，缺乏对历史负载基线的趋势预测能力。
• F-12：尚未实装自动扩缩容（Auto-Scaling）策略，边缘节点无法根据集群总体请求量自动执行物理停机休眠或唤醒，存在电力损耗优化空间。

### 1. 质量保障与自动化测试体系 (Testing & QA)

#### T-01 单元测试规范 (P0 级强制)
• 目标：必须确保核心模块（探针、调度器、数据模型）的独立正确性，行覆盖率底线目标必须设定为 ≥80%。
• 技术栈：后端必须采用 pytest + pytest-asyncio + pytest-mock。
• 规范示例 (探针与 Redis 状态更新)：
```python
import pytest
from unittest.mock import patch, MagicMock
from topology_sentinel import check_mount

def test_check_mount_exists():
    with patch('os.path.exists', return_value=True):
        assert check_mount('/mnt/media') is True

@pytest.mark.asyncio
async def test_redis_state_update():
    mock_redis = MagicMock()
    with patch('topology_sentinel.get_redis_client', return_value=mock_redis):
        from topology_sentinel import update_state
        await update_state('media_engine', 'OFF')
        mock_redis.set.assert_called_once_with('media_engine', 'OFF')
```

#### T-02 集成测试规范 (P1 级强制)
• 目标：验证容器协同与硬件级熔断降级。必须在隔离的 Docker Compose 环境中运行完整链路。
• 规范示例 (模拟存储拔出触发的三步优雅降级)：
```python
import requests
import time
import subprocess

def test_storage_loss_triggers_fuse_sequence():
    # 验证初始在线状态
    resp = requests.get("http://zen70-main/api/v1/capabilities")
    assert resp.json()["services"]["media"]["status"] == "online"

    # 模拟物理存储拔出
    subprocess.run(["sudo", "umount", "/mnt/media"])
    time.sleep(20) # 等待探针滑动窗口防抖

    # 1. 必须验证 API 层是否已成功熔断 (红线：严禁不熔断直接操作容器)
    fuse_resp = requests.get("http://zen70-main/api/v1/media/stream")
    assert fuse_resp.status_code == 503

    # 2. 必须验证网关状态是否更新
    resp = requests.get("http://zen70-main/api/v1/capabilities")
    assert resp.json()["services"]["media"]["status"] == "offline"

    # 3. 必须验证底层容器是否被正确强制冻结/停止
    container_status = subprocess.run(["docker", "inspect", "--format", "{{.State.Status}}", "jellyfin"], capture_output=True, text=True)
    assert container_status.stdout.strip() in ["paused", "exited"]
```

#### S-06 混沌工程注入 (Chaos Engineering)
• 目标：建议主动注入网络分区、容器宕机等故障，验证系统自愈能力。作为高级可靠性保障手段，MVP 阶段可通过提交 ADR 暂缓实施，避免过度设计拖慢交付。
• 规范示例 (随机容器熔断与网络丢包)：
```python
#!/usr/bin/env python3
import subprocess
import time
import random

def kill_random_container():
    container = random.choice(['jellyfin', 'frigate', 'ollama'])
    subprocess.run(['docker', 'kill', container], check=False)
    time.sleep(10)
    # 验证 Docker restart policy 是否生效
    status = subprocess.run(['docker', 'inspect', '--format', '{{.State.Status}}', container], capture_output=True, text=True)
    if status.stdout.strip() == 'running':
        print(f"✅ {container} auto-restarted")

def network_partition():
    # 模拟 50% 底层丢包，验证探针心跳超时机制
    subprocess.run(['tc', 'qdisc', 'add', 'dev', 'eth0', 'root', 'netem', 'loss', '50%'])
```

#### C-06 标准化发布流程 (Release Pipeline)
• 目标：必须规范化 Semantic Versioning (语义化版本)，严禁人工手动打 Tag。一切版本发布必须由自动化脚本驱动。
• 发布脚本 (release.sh) 准则：
```bash
#!/bin/bash
set -e

# 必须强制检查工作区纯净度
if [[ -n $(git status -s) ]]; then
    echo "Error: Working directory not clean."
    exit 1
fi

# 自动递增版本号 (Patch/Minor/Major)
CURRENT_VERSION=$(cat VERSION)
NEW_VERSION=$(echo $CURRENT_VERSION | awk -F. '{$NF+=1; print $1"."$2"."$NF}')
echo $NEW_VERSION > VERSION

# 修改对应配置字典
sed -i.bak "s/\"version\": \".*\"/\"version\": \"$NEW_VERSION\"/" frontend/package.json

# 自动生成 Tag 与镜像
git commit -m "chore(release): prepare version $NEW_VERSION"
git tag -a "v$NEW_VERSION" -m "Release version $NEW_VERSION"
docker build -t $REGISTRY/zen70-backend:v$NEW_VERSION ./backend
```
